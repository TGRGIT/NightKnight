# Requesting entitlements: Time-Sensitive Notifications & CarPlay

Two capabilities NightKnight wants follow very different paths:

- **Time-Sensitive Notifications** is *self-service* — you enable it yourself, no
  Apple review. Needed so a glucose alarm's `.timeSensitive` level can break through
  Focus / Do Not Disturb.
- **CarPlay** requires *Apple approval* — you submit a request, describe the app, and
  wait for Apple to grant the entitlement before you can build against it (or ship).

**Prerequisites for both**

- Paid Apple Developer Program membership.
- Signed in as the **Account Holder** or an **Admin** (identifier/capability changes
  need that role).
- App bundle id: `be.cooney.nightknight.NightKnight`.

> **This project is generated by XcodeGen.** `ios/NightKnight/NightKnight.entitlements`,
> the `Info.plist`s and the `.xcodeproj` are all produced from **`ios/project.yml`**.
> Declare entitlements/Info-plist keys **there** (under a target's `entitlements:` /
> `info:` `properties:`), then `cd ios && xcodegen generate`. Adding a capability by hand
> in the `.entitlements` file or via Xcode's **Signing & Capabilities** UI works until the
> next `xcodegen generate`, which **overwrites it**. (Enabling the capability on the *App
> ID* in the portal is separate and is not regenerated — see each part below.)

---

## Part A — Time-Sensitive Notifications (self-service, ~5 min)

This lets a notification with `interruptionLevel = .timeSensitive` pierce Focus / Do Not
Disturb. There is **no Apple review** — it's a capability you just turn on. NightKnight's
alarms already set `.timeSensitive` (`ios/NightKnight/AlarmManager.swift`).

Two things have to line up: the **entitlement** in the build, and the matching
**capability on the App ID** so code-signing accepts it.

### 1. Declare the entitlement in `project.yml`

Under `targets: → NightKnight: → entitlements: → properties:` add:

```yaml
        com.apple.developer.usernotifications.time-sensitive: true
```

then `cd ios && xcodegen generate`. (The committed `NightKnight.entitlements` already
carries this key once the iOS-alarms work is in; keeping it in `project.yml` is what makes
regeneration reproduce it.)

### 2. Enable the capability on the App ID

**Easiest — let Xcode manage it:** open the project → **NightKnight** target →
**Signing & Capabilities**, keep **Automatically manage signing** on, **+ Capability →
Time Sensitive Notifications**. Xcode enables it on the App ID and regenerates the
provisioning profile. (This also writes the key into `.entitlements`, but that's fine —
it matches what `project.yml` generates.)

**Or via the portal (manual signing):** <https://developer.apple.com/account> →
**Certificates, Identifiers & Profiles → Identifiers →** `be.cooney.nightknight.NightKnight`,
tick **Time Sensitive Notifications**, **Save**, then regenerate and install the profile.

### 3. Verify

- Build to a device, turn on a Focus mode, trigger an alarm — it should still alert.
- The app then appears under **Settings → Focus → [a Focus] → Allowed Notifications**, and
  the user can allow/deny "Time Sensitive" alerts in the app's notification settings. The
  entitlement grants the *ability*; the user can still opt out.

> **Not** the same as **Critical Alerts**
> (`com.apple.developer.usernotifications.critical-alerts`), which plays a loud sound even
> when the phone is muted and **does** require an Apple approval request (similar to
> CarPlay below). NightKnight deliberately does not use Critical Alerts.

## Part B — CarPlay (requires Apple approval)

CarPlay entitlements are gated. You request one, describe the app, and are approved before
the entitlement is added to your account. You can't self-enable it, and the app won't
appear in CarPlay without it.

| | |
|---|---|
| **Category to request** | **Driving Task** |
| **Entitlement key** | `com.apple.developer.carplay-driving-task` |
| **Where to request** | <https://developer.apple.com/contact/carplay/> |
| **What ships in CarPlay** | A glanceable current-glucose + trend screen (`CPInformationTemplate`) |
| **Typical turnaround** | Days to a few weeks, with follow-up questions; not guaranteed |

### 1. Choose the right app category

CarPlay only admits apps that fit one of Apple's defined categories, each with its own
entitlement key:

| Category | Entitlement key | Fit for NightKnight |
|---|---|---|
| **Driving Task** | `com.apple.developer.carplay-driving-task` | **Best fit** — glanceable, short tasks a driver would otherwise reach for the phone to do |
| Navigation | `com.apple.developer.carplay-navigation` | No |
| Audio | `com.apple.developer.carplay-audio` | No |
| Communication | `com.apple.developer.carplay-communication` | No |
| EV Charging / Parking / Fueling / Quick Ordering | `com.apple.developer.carplay-charging`, `-parking`, `-fueling`, `-quick-ordering` | No |

For a glanceable "current glucose + trend + recent history" screen, request the
**Driving Task** category. There is no general "show a number" entitlement; Driving Task
is the closest legitimate fit and the one Apple steers glanceable utilities toward.

**Reality check.** Apple reviews CarPlay requests strictly for driver distraction and
favours established use cases. A CGM display is *plausible* under Driving Task but not
guaranteed — approval is at Apple's discretion. Frame the request around **safety and
glanceability**: a read-only display that *reduces* the temptation to pick up the phone,
no interaction needing sustained attention, and NightKnight is a personal-health follower
(explicitly **not** a medical device).

### 2. (Recommended) Prototype in the simulator first

You can build the CarPlay scene in the **simulator before approval**: add the entitlement
key to your local build and run — simulator builds aren't profile-validated, so the
CarPlay display connects without Apple having granted anything. This is the fastest way to
produce the **screenshots/mockups Apple's request form asks for**. (You still can't run on
a real device or ship until the entitlement is granted.)

To open the simulated head unit: run the app, then in the Simulator menu
**I/O → External Displays → CarPlay**. The `-NKDemo` launch argument
(`ios/Shared/DemoData.swift`) fills it with realistic data and no server.

### 3. Submit the request

Go to **<https://developer.apple.com/contact/carplay/>**, sign in, choose app type
**Driving Task**, and answer using the packet below. Attach the mockup from step 2. Apple
replies by email; expect some back-and-forth.

**Ready-to-paste packet** (edit the bracketed bits; keep the safety language — it's what's
actually reviewed):

- **App name:** NightKnight
- **Bundle ID:** `be.cooney.nightknight.NightKnight`
- **App Store URL:** [link, or "In TestFlight / pre-release"]
- **CarPlay app category:** Driving Task
- **What does your app do?**
  > NightKnight is a private follower/viewer for continuous glucose monitoring (CGM) data.
  > It connects to the user's own NightKnight or Nightscout-compatible server and shows
  > their current glucose value, trend direction and how long ago it was read, plus
  > optional high/low alerts. It is not a medical device; it presents information the user
  > already owns.
- **What will it do in CarPlay, specifically?**
  > A single glanceable screen: the latest glucose value, a colour/word status (e.g. "In
  > range"), the trend arrow, and the reading's age. It refreshes automatically on the CGM
  > cadence (~5 min). Optionally, an out-of-range reading raises a standard CarPlay alert
  > with audio so the driver is told to check their phone or pull over — without reading
  > anything detailed while moving.
- **Why a Driving Task app?**
  > Checking current glucose is a short, important task a driver would otherwise do by
  > picking up and unlocking their phone — a known distraction. Showing only the current
  > value and trend on the car's display lets them glance once, like checking fuel or
  > speed, and keep their attention on the road.
- **How have you minimised driver distraction?**
  > One screen, no scrolling, no charts, no interaction required — it just displays. Large,
  > high-contrast value using CarPlay's own templates. No history, graphs, data entry or
  > settings in the car (all configured on the phone beforehand). Updates are automatic;
  > the driver never taps to refresh. Out-of-range states use a brief standard alert rather
  > than detail to read. Fully compliant with Apple's CarPlay App Design and HIG for
  > Driving Task apps.
- **Audience / usage:** [e.g. "People with diabetes who self-host their CGM data; N TestFlight users."]

**The mockup:** one ~800×480 landscape frame of a `CPInformationTemplate`-style layout —
title "Glucose", then large rows: **`124 mg/dL`** (colour by range), **`→ Steady`**,
**`Updated 1 min ago`**, dark theme. Reuse the look of the dashboard's current-card
(`ios/NightKnight/DashboardView.swift`), or capture it from the CarPlay Simulator (step 2).

### 4. After approval

Apple adds the CarPlay entitlement to your developer account. Then:

1. Developer portal → **Identifiers →** `be.cooney.nightknight.NightKnight` → enable the
   granted **CarPlay** capability. **Save.**
2. Regenerate the app's provisioning profile so it includes the entitlement; install it.
3. Add the entitlement in **`project.yml`** (not the `.entitlements` file directly — see
   the XcodeGen note up top), under `targets: → NightKnight: → entitlements: → properties:`:
   ```yaml
        com.apple.developer.carplay-driving-task: true
   ```
   then `cd ios && xcodegen generate`.
4. (Manual signing) confirm the profile the build uses carries it.

### 5. Implement the CarPlay scene

CarPlay apps render Apple's **templates** — you can't draw arbitrary UI. A Driving Task app
gets `CPInformationTemplate`, `CPListTemplate`, `CPGridTemplate`, `CPPointOfInterestTemplate`,
`CPTabBarTemplate` and alerts. A glanceable glucose readout is a `CPInformationTemplate`.

**a. Declare the scene** in `project.yml` under `targets: → NightKnight: → info: → properties:`:

```yaml
        UIApplicationSceneManifest:
          UIApplicationSupportsMultipleScenes: true
          UISceneConfigurations:
            CPTemplateApplicationSceneSessionRoleApplication:
              - UISceneConfigurationName: NightKnight-CarPlay
                UISceneDelegateClassName: $(PRODUCT_MODULE_NAME).CarPlaySceneDelegate
```

**b. Add `ios/NightKnight/CarPlaySceneDelegate.swift`** — it reuses the existing
`APIClient`, `Settings`, `ReadingCache`, `GlucoseValue`/`GlucoseBand` and `CurrentReading`,
so there's no new data layer:

```swift
import CarPlay

final class CarPlaySceneDelegate: UIResponder, CPTemplateApplicationSceneDelegate {
    private var interfaceController: CPInterfaceController?
    private var refreshTask: Task<Void, Never>?

    func templateApplicationScene(_ scene: CPTemplateApplicationScene,
                                  didConnect interfaceController: CPInterfaceController) {
        self.interfaceController = interfaceController
        let template = CPInformationTemplate(title: "Glucose", layout: .leading,
                                             items: [], actions: [])
        interfaceController.setRootTemplate(template, animated: false, completion: nil)
        startRefreshing(template)
    }

    func templateApplicationScene(_ scene: CPTemplateApplicationScene,
                                  didDisconnect interfaceController: CPInterfaceController) {
        refreshTask?.cancel()
        self.interfaceController = nil
    }

    /// Glanceable only: current value, status, trend, age. Refreshes on CGM cadence.
    private func startRefreshing(_ template: CPInformationTemplate) {
        refreshTask?.cancel()
        refreshTask = Task { @MainActor in
            while !Task.isCancelled {
                let reading = (try? await APIClient(settings: .shared).current())
                    ?? ReadingCache.load()            // warm fallback the app/widget keep fresh
                template.items = items(for: reading)
                try? await Task.sleep(for: .seconds(60))
            }
        }
    }

    private func items(for reading: CurrentReading?) -> [CPInformationItem] {
        guard let r = reading else {
            return [CPInformationItem(title: "Glucose", detail: "Open NightKnight on your phone")]
        }
        let unit = Settings.shared.preferredUnit
        let band = GlucoseBand.of(mgdl: r.value.mgdl)
        let age = Int(Date().timeIntervalSince(r.date) / 60)
        return [
            CPInformationItem(title: "\(r.value.display(in: unit)) \(unit.label)", detail: band.label),
            CPInformationItem(title: "Trend", detail: "\(r.trend.glyph)  \(r.trendLabel)"),
            CPInformationItem(title: "Updated", detail: age <= 0 ? "just now" : "\(age) min ago"),
        ]
    }
}
```

**c. Keep it glanceable.** Large value, trend, age — no scrolling charts, no input that
needs attention; this is what review judges. The 24h chart, AGP and episodes stay
phone-only by design (and by policy). Background freshness already exists: the widget/watch
keep `ReadingCache` warm and the app schedules background refreshes (`BackgroundRefresh` in
`NightKnightApp.swift`). For out-of-range alerting in the car, present a brief
`CPAlertTemplate` from the existing `AlarmManager` evaluation rather than anything to read.

### 6. Test in the CarPlay Simulator

Run on an iOS Simulator, then **Xcode → Open Developer Tool → Simulator**, and in the
Simulator menu **I/O → External Displays → CarPlay**. Your `CPInformationTemplate` renders;
`-NKDemo` populates it without a server. Then test on a real head unit once the device
profile carries the entitlement.
